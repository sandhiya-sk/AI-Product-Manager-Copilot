# AI Product Manager Copilot — Modules 1–3: Detailed Implementation Specification

---

## Overview

The **AI Product Manager Copilot** is a full-stack web application that helps product managers collect, process, and analyze product feedback from both customers and internal stakeholders. This document provides an exact and exhaustive implementation blueprint for the first three modules:

| Module | Name | Role |
|--------|------|------|
| Module 1 | User Interface & Authentication | React frontend with role-based access (PM / Customer) |
| Module 2 | Data Ingestion Service | CSV upload, text form ingestion, validation, JSON conversion, raw DB storage |
| Module 3 | Data Processing & Preprocessing Pipeline | Semantic deduplication, text cleaning, tokenization, lemmatization, processed DB storage |

**Technology Stack:**
- **Backend:** Python (Flask 3.0), REST API
- **Frontend:** React 18 (Vite), React Router v6, Axios, plain CSS
- **Primary Database:** PostgreSQL (tables: `users`, `raw_feedback`, `processed_feedback`)
- **NLP Libraries:** `spaCy` (en_core_web_sm), `NLTK`, `sentence-transformers` (all-MiniLM-L6-v2)
- **CSV Parsing:** `pandas`
- **Authentication:** JWT via `flask-jwt-extended`
- **ORM:** `SQLAlchemy 2.0` with `psycopg2`
- **Environment Config:** `python-dotenv`

## Module 1 — User Interface & Authentication

### 1.1 Purpose

Module 1 provides the React-based web frontend that serves two distinct user roles:
1. **Product Manager (PM):** Can upload CSV files containing bulk customer feedback, submit structured text-based feedback forms, and manage projects.
2. **Customer:** Can submit structured text-based feedback forms only (no CSV upload capability).

### 1.2 Role-Based Access Control (RBAC)

Two roles are enforced at both frontend (route guards) and backend (JWT claim validation):
- `role: "product_manager"` — can access CSV upload panel, text form, and dashboard
- `role: "customer"` — can only access the text feedback submission form

Each user is issued a JWT on login. The JWT payload contains:
```json
{
  "user_id": "<uuid>",
  "email": "<email>",
  "role": "product_manager | customer",
  "project_id": "<project_uuid>",
  "iat": "<timestamp>",
  "exp": "<timestamp>"
}
```

### 1.3 Frontend Pages & Components

#### Pages

| Page | Route | Accessible By |
|------|--------|----------------|
| Login | `/login` | All |
| Register | `/register` | All |
| PM Dashboard | `/dashboard` | product_manager |
| CSV Upload | `/upload/csv` | product_manager |
| Text Feedback Form | `/upload/feedback` | product_manager, customer |
| Processing Status | `/status` | product_manager |

#### Key Components

1. **`<LoginForm />`** — Email + password fields, calls `POST /api/auth/login`, stores JWT in `localStorage`.
2. **`<RegisterForm />`** — Registers user with role selection. Calls `POST /api/auth/register`.
3. **`<ProtectedRoute />`** — HOC that reads JWT from `localStorage`, decodes it, and redirects unauthorized users. Accepts a `roles` prop for role-based filtering.
4. **`<CSVUploadPanel />`** — Only rendered for `product_manager` role. Accepts a `.csv` file via drag-and-drop or file picker. Validates MIME type on the frontend before sending. Calls `POST /api/ingest/csv`.
5. **`<FeedbackForm />`** — Structured text input form with fields:
   - `subject` (text input, required)
   - `description` (textarea, required)
   - `priority` (dropdown: Low / Medium / High / Critical)
   - `category` (dropdown: Bug / Feature Request / Improvement / Complaint / General)
   - `tags` (comma-separated text input, parsed to list client-side)
   - `product_name` (text input)
   - `product_version` (text input)
   - `sentiment_self_reported` (optional dropdown: Positive / Negative / Neutral)
   - Sends JSON to `POST /api/ingest/feedback`
6. **`<StatusPanel />`** — Polls `GET /api/status/{feedback_id}` every 5 seconds to show processing status with a visual progress indicator.
7. **`<Navbar />`** — Role-aware navigation bar with links conditional on user role. Shows logout button.
8. **`<AuthContext />`** — React context that provides `user`, `token`, `login()`, and `logout()` to the entire app.

## Module 2 — Data Ingestion Service

### 2.1 Purpose

Module 2 is the ingestion layer. It accepts:
1. **CSV files** from Product Managers (with a fixed, predefined schema).
2. **Structured text form submissions** from both Product Managers and Customers.

All inputs are validated, parsed, normalized, converted to a canonical JSON structure, and stored in the **PostgreSQL Raw Feedback Database** (`raw_feedback` table).

### 2.2 CSV Fixed Schema

All CSVs uploaded by product managers **must** conform to this exact column schema (order-independent, case-insensitive header matching):

| Column Name | Data Type | Required | Description |
|-------------|-----------|----------|-------------|
| `feedback_id` | String | No | If absent, system generates UUID |
| `customer_name` | String | No | Name of the customer |
| `customer_email` | String | No | Email of the customer |
| `subject` | String | **Yes** | Short title of the feedback |
| `description` | String | **Yes** | Full text body of the feedback |
| `priority` | Enum string | No | Low / Medium / High / Critical |
| `category` | Enum string | No | Bug / Feature Request / Improvement / Complaint / General |
| `product_name` | String | No | Product being reviewed |
| `product_version` | String | No | Version of the product |
| `submission_date` | Date | No | Date of original submission (YYYY-MM-DD) |
| `tags` | String (comma-separated) | No | e.g., "login,mobile,performance" |
| `sentiment_self_reported` | Enum string | No | Positive / Negative / Neutral |

**Validation Rules (enforced by Flask before storage):**
- File must be `.csv` (MIME type `text/csv` or `application/vnd.ms-excel`)
- File size <= 10 MB
- File must not be empty (at least 1 data row after header)
- Columns `subject` and `description` must be present and non-null for every row
- `priority` must be one of the enum values (default to `Medium` if absent/invalid)
- `category` must be one of the enum values (default to `General` if absent/invalid)
- Each row is validated individually; invalid rows are collected and returned in `failed_rows`

Both PM and Customer text submissions share the same field structure:

### 2.6 Batch Processing

When a CSV with multiple rows is uploaded:
- A `batch_id` (UUID4) is generated for the entire upload session.
- All rows share the same `batch_id`, stored in the `raw_metadata` JSONB column.
- This allows querying all records from a single upload: `WHERE raw_metadata->>'batch_id' = '<batch_id>'`

Functions:
- `validate_csv_file(file_storage)` → `(bool, list[str])` — validates MIME, size, emptiness
- `validate_csv_schema(df)` → `(bool, list[str])` — validates required columns exist

## Module 3 — Data Processing & Preprocessing Pipeline

### 3.1 Purpose

Module 3 is a background processing pipeline triggered by a Product Manager via API or automatically after ingestion. It:
1. Fetches all `raw_feedback` records with `processing_status = 'pending'`
2. Performs **Semantic Duplicate Detection** using sentence embeddings
3. Applies **Text Cleaning** to remove noise
4. Applies **Standardization** (case, date, priority, status normalization)
5. Performs **Tokenization** (NLTK)
6. Performs **Lemmatization** (spaCy)
7. Generates **Processing Metadata** (word count, char count, timestamps)
8. Stores results in the **`processed_feedback`** PostgreSQL table

### 3.2 Semantic Duplicate Detection — Detailed Algorithm

**Libraries:** `sentence-transformers` (model: `all-MiniLM-L6-v2`), `numpy`, `scipy`

**Step 1: Fetch Pending Records**
```python
records = db.execute(
    "SELECT * FROM raw_feedback WHERE processing_status = 'pending' ORDER BY upload_timestamp ASC"
)
```

**Step 2: Generate Sentence Embeddings**
- Concatenate `subject + " " + description` for each record to form `full_text`.
- Encode all `full_text` values using `SentenceTransformer('all-MiniLM-L6-v2').encode()`.
- Result: a 2D NumPy array of shape `(N, 384)` — one 384-dim vector per record.

**Step 3: Compute Pairwise Cosine Similarity**
- Use `scipy.spatial.distance.cosine(a, b)` → cosine distance (1 - similarity).
- `similarity = 1 - cosine_distance`
- Threshold: `SIMILARITY_THRESHOLD = 0.85` (from environment variable)
- If `similarity >= 0.85` → semantically duplicate.

**Step 4: Union-Find Grouping**
- Implement a Union-Find (Disjoint Set Union) data structure.
- For every pair `(i, j)` where `similarity(i, j) >= 0.85`, merge them into the same component.
- After all pairs are processed, extract the final groups using `find()`.

**Step 5: Assign Canonical Records and Weights**
For each group:
- Sort members by `upload_timestamp` ascending → oldest record = **canonical record**
- Canonical record gets:
  - `duplicate_group_id = uuid4()` (new UUID for this group)
  - `weight = len(group)` (total number of semantically similar records)
  - `processing_status = 'processing'`
- All other records in the group (duplicates) get:
  - `duplicate_group_id = same group UUID`
  - `processing_status = 'duplicate'`
- For records with no duplicates (singleton groups):
  - `duplicate_group_id = uuid4()` (its own unique group)
  - `weight = 1`

**Step 6: Update raw_feedback Table**
```sql
UPDATE raw_feedback
SET weight = %s,
    duplicate_group_id = %s,
    processing_status = %s
WHERE feedback_id = %s;
```

### 3.3 Text Cleaning — Applied to canonical records' `description` field

| Step | Operation | Code |
|------|-----------|------|
| 1 | Remove extra whitespace | `re.sub(r'\s+', ' ', text).strip()` |
| 2 | Remove HTML tags | `re.sub(r'<[^>]+>', '', text)` |
| 3 | Remove URLs | `re.sub(r'https?://\S+|www\.\S+', '', text)` |
| 4 | Remove emojis | Filter chars where `unicodedata.category(c)` starts with 'So' or 'Sm' |
| 5 | Normalize punctuation | `re.sub(r'[!]{2,}', '!', re.sub(r'\.{2,}', '.', text))` |
| 6 | Remove special chars | `re.sub(r"[^\w\s.,!?'\"\\-]", '', text)` |

### 3.4 Standardization

| Operation | Rule |
|-----------|------|
| Lowercase | `text.lower()` applied to cleaned description |
| Priority normalization | Map variants (e.g., "urgent", "p0", "high priority") → `High`; "p2", "low" → `Low`; else → `Medium` |
| Category normalization | Case-insensitive fuzzy match to enum list; unrecognized → `General` |
| Date normalization | `dateutil.parser.parse(date_str).strftime('%Y-%m-%d')` wrapped in try/except |
| Status normalization | Only allow `pending`, `processing`, `processed`, `failed`, `duplicate` |

### 3.5 Tokenization — Using NLTK

```python
import nltk
from nltk.tokenize import word_tokenize
from nltk.corpus import stopwords

stop_words = set(stopwords.words('english'))

def tokenize(text: str) -> list[str]:
    tokens = word_tokenize(text)
    tokens = [
        t.lower() for t in tokens
        if t.isalpha() and t.lower() not in stop_words
    ]
    return tokens
```

Output stored as PostgreSQL `TEXT[]` array.

### 3.6 Lemmatization — Using spaCy

```python
import spacy
nlp = spacy.load("en_core_web_sm")

def lemmatize(tokens: list[str]) -> list[str]:
    doc = nlp(" ".join(tokens))
    lemmas = [
        token.lemma_.lower()
        for token in doc
        if not token.is_stop and not token.is_punct and token.lemma_ != "-PRON-"
    ]
    return lemmas
```

Output stored as PostgreSQL `TEXT[]` array.

### 3.9 Pipeline Orchestration — `services/processing_pipeline.py`

```python
class ProcessingPipeline:

    def __init__(self):
        self.model = SentenceTransformer('all-MiniLM-L6-v2')
        self.nlp = spacy.load('en_core_web_sm')
        self.threshold = float(os.getenv('SIMILARITY_THRESHOLD', '0.85'))

    def run(self, project_id=None) -> dict:
        records = self.fetch_pending(project_id)
        if not records:
            return {"processed": 0, "duplicates": 0, "failed": 0}

        groups = self.detect_duplicates(records)
        processed, duplicates, failed = 0, 0, 0

        for group in groups:
            canonical = group['canonical']
            dups = group['duplicates']

            try:
                cleaned = self.clean_text(canonical['description'])
                standardized = self.standardize(cleaned, canonical)
                tokens = self.tokenize(standardized)
                lemmas = self.lemmatize(tokens)
                metadata = self.build_metadata(canonical, cleaned, tokens, lemmas)
                self.insert_processed(canonical, cleaned, standardized, tokens, lemmas, metadata, group['group_id'])
                self.update_raw_status(canonical['feedback_id'], 'processed')
                processed += 1
            except Exception as e:
                self.update_raw_status(canonical['feedback_id'], 'failed')
                failed += 1

            for dup in dups:
                self.update_raw_status(dup['feedback_id'], 'duplicate', group['group_id'])
                duplicates += 1

        return {"processed": processed, "duplicates": duplicates, "failed": failed}
```

### 3.10 Triggering Strategy

The pipeline is triggered in three ways:

1. **On-demand via API:** `POST /api/process/run` runs the pipeline in a background thread (`threading.Thread`).
2. **Auto-trigger post-ingestion:** After each successful CSV upload or form submission, `run_pipeline_async()` is called if there are more than `PIPELINE_AUTO_TRIGGER_THRESHOLD` (default: 5) pending records.
3. **Scheduled (optional):** Using `APScheduler` every 10 minutes in production.

def tokenize(text: str) -> list[str]:
    tokens = word_tokenize(text)
    return [t.lower() for t in tokens if t.isalpha() and t.lower() not in stop_words]

nlp = spacy.load('en_core_web_sm')

def lemmatize(tokens: list[str]) -> list[str]:
    doc = nlp(' '.join(tokens))
    return [token.lemma_.lower() for token in doc if not token.is_stop and not token.is_punct and len(token.lemma_) > 1]
